Browser Only mode — the simplest path. I pick a DICOM file and it renders immediately. No server call, no upload. Everything happens right in the browser.

Behind the scenes, Cornerstone3D — the same open-source engine used in professional medical imaging software — is parsing the raw DICOM binary and rendering it on screen. The file never leaves your computer.

On the right — the metadata panel. Patient name, Modality, Study Date, Manufacturer. All extracted from the DICOM file's embedded tags. DICOM isn't just an image — it's a clinical record. Every file carries context alongside the pixels.

Important production note: this mode works with the original DICOM file — full pixel fidelity, all metadata, real-time brightness and contrast adjustments. The trade-off is file size — a dental X-ray runs one to ten megabytes, an MRI series up to fifty. We'll see the lighter alternative in Server mode.

Server Based mode — same file, but now watch the Network tab. Two calls: a POST sending the DICOM to our .NET backend, then a GET fetching back a rendered PNG.

The backend — .NET 8 with fo-dicom — parses the DICOM, extracts metadata, renders each frame to PNG, and returns URLs. Look at the sizes: original DICOM was about eight megabytes, the PNG coming back is a few hundred kilobytes.

The trade-off: PNG is fast and lightweight — great for slow connections. But you lose the original sixteen-bit pixel depth, which means no real-time brightness and contrast control in the browser. Metadata is preserved separately in the JSON response.

For production, the right approach is hybrid — store the original DICOM so you never lose fidelity, serve a compressed preview for fast loading. We could push even further with lossy JPEG or WebP — down to twenty kilobytes per frame, instant on any connection. The original stays on the server for diagnosis, AI analysis, or measurements.

This backend is also where future AI capabilities plug in — cavity detection, automated measurements, report generation. The architecture is already here.

These are the three PACS integration protocols. They need the native shell to talk directly to imaging servers without browser security restrictions.



DICOMweb Poll — Copilot asks the clinic "any new X-rays?" every two seconds.

DICOMweb is the modern HTTP-based imaging standard, introduced around 2015. About fifty to sixty percent of modern PACS systems support it and growing. What makes it attractive: the clinic changes nothing. We just need their server's URL — one setting in Copilot. Zero IT involvement at the clinic.

This URL at the bottom is our Gateway server — standing in for the clinic's PACS. I'll open it, upload a file — simulating a tech taking an X-ray.

Close the window and — there it is. The study appeared automatically. No refresh, no click. Copilot's polling loop detected it within two seconds. I click it, the app fetches the image via WADO-RS — a DICOMweb standard for retrieving pixel data — and renders it in the same Cornerstone3D viewer.

In production, swap this URL with a clinic's real PACS address. Same flow, no code changes.



Now the push side — the clinic sends images to Copilot instead of Copilot asking for them.

DIMSE C-STORE has been the backbone of medical imaging since 1993. Runs over raw TCP. Every imaging device ever made supports it — one hundred percent coverage. STOW-RS is the modern alternative — same push concept, but over HTTPS. About fifty to sixty percent coverage today.

Both need a one-time clinic setup — the IT admin adds Copilot as a destination. Three fields for DIMSE, one URL for STOW-RS. Two minutes, done once. After that, every X-ray flows automatically.

In our demo, this URL is the Vendor Simulator — a second server standing in for the clinic's imaging software. When I upload here, a script pushes the image to Copilot's Gateway via both protocols simultaneously.

I'll upload a file. A few seconds for the script to run — and there it is in the DIMSE tab, tagged "DIMSE C-STORE TCP." Now switch to STOW-RS — same image, already here, tagged "STOW-RS HTTP."

One upload, two protocols, proven side by side.
